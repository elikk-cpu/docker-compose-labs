# RESULT — Project 01

## Краткий итог

Я запустил контейнер Nginx со статическим сайтом на порту 8088.

Контейнер:
- использует image nginx:1.30.4-alpine3.24;
- называется landing;
- подключён к пользовательской сети landing_net;
- доступен с хоста через http://localhost:8088;
- доступен из другого контейнера по DNS-имени landing;
- получает сайт и конфигурацию через read-only bind mounts.

## Что я создал или изменил

Новые Dockerfile или Compose-файлы в этом проекте не создавались.

Я:
- скачал image Nginx;
- создал сеть landing_net;
- запустил контейнер landing;
- подключил каталог site/;
- подключил файл nginx/default.conf;
- проверил сеть, mounts, логи и использование ресурсов;
- воспроизвёл две неисправности;
- подготовил каталог evidence/ с результатами проверок.

## Архитектура и принятые решения

Ubuntu VM:8088
      |
      v
container landing:8080
      |
      +-- /usr/share/nginx/html
      |      bind mount: ./site, read-only
      |
      +-- /etc/nginx/conf.d/default.conf
             bind mount: ./nginx/default.conf, read-only

Контейнер подключён к пользовательской bridge-сети landing_net.

Внутри этой сети другие контейнеры могут обращаться к Nginx по адресу:

http://landing:8080

Я использовал read-only mounts, чтобы контейнер не мог изменить сайт и конфигурацию.

## Команды запуска и остановки

Создание сети:

docker network create landing_net

Запуск контейнера:

docker run -d \
  --name landing \
  --network landing_net \
  -p 8088:8080 \
  --mount type=bind,src="$(pwd)/site",dst=/usr/share/nginx/html,readonly \
  --mount type=bind,src="$(pwd)/nginx/default.conf",dst=/etc/nginx/conf.d/default.conf,readonly \
  nginx:1.30.4-alpine3.24

Штатная остановка и повторный запуск существующего контейнера:

docker stop landing
docker start landing

Удаление и полное пересоздание контейнера:

docker stop landing
docker rm landing

После удаления контейнер был создан заново командой docker run.

## Ошибки и root cause

### Поломка 1. Контейнер сразу завершался

Я запустил image с командой:

echo "wrong command"

Эта команда заменила основной процесс Nginx.

echo выполнилась и сразу завершилась, поэтому контейнер перешёл в состояние exited.

Root cause: контейнер существует, пока работает его основной процесс PID 1. После завершения основного процесса контейнер останавливается.

### Поломка 2. Неправильный опубликованный порт

Я запустил контейнер с сопоставлением:

8089:80

Но предоставленная конфигурация Nginx слушает порт:

8080

Контейнер находился в состоянии running, а логи показывали успешный запуск Nginx, но запрос к localhost:8089 завершался ошибкой.

Root cause: host port был направлен на порт 80 контейнера, на котором в данном проекте ничего не слушало.

Правильное сопоставление:

8088:8080

## Best practices, которые я применил

- Использовал конкретный tag nginx:1.30.4-alpine3.24, а не latest.
- Создал отдельную пользовательскую bridge-сеть.
- Использовал DNS-имя контейнера вместо IP-адреса.
- Подключил сайт и конфигурацию в read-only режиме.
- Опубликовал только необходимый порт.
- Не использовал --privileged.
- Не изменял конфигурацию вручную внутри контейнера.
- Для диагностики использовал ps, logs, inspect, history и stats.
- Использовал штатную остановку через docker stop.

## Ответы на диагностические вопросы

### Чем image отличается от container?

Image — неизменяемый шаблон с файловой системой и настройками приложения.

Container — запущенный или остановленный экземпляр image со своим writable layer.

### Что хранится во writable layer?

Изменения файловой системы, сделанные внутри конкретного контейнера и не находящиеся в mount.

После удаления контейнера его writable layer удаляется.

### Почему localhost не указывает на соседний контейнер?

localhost внутри контейнера указывает на этот же контейнер.

Для обращения к соседнему контейнеру в пользовательской Docker-сети используется его DNS-имя, например landing.

### Почему внутренний порт 8080, а внешний 8088?

Nginx слушает 8080 внутри контейнера.

Docker публикует его на порту 8088 хоста через mapping:

8088:8080

Левая часть — порт хоста, правая — порт контейнера.

### Чем bind mount отличается от volume?

Bind mount подключает конкретный файл или каталог файловой системы хоста.

Volume управляется Docker и обычно лучше подходит для постоянных данных приложений и баз данных.

### Почему read-only mount безопаснее?

Контейнер может читать файлы, но не может случайно или намеренно изменить сайт или конфигурацию.

### Что показывают диагностические команды?

- docker logs — вывод основного процесса контейнера;
- docker inspect — подробную конфигурацию контейнера;
- docker history — слои image и команды их создания;
- docker stats — использование CPU, RAM, сети и диска.

### Почему docker stop предпочтительнее docker kill?

docker stop сначала отправляет процессу сигнал SIGTERM и даёт время корректно завершиться.

docker kill обычно немедленно отправляет SIGKILL, не позволяя приложению выполнить штатное завершение.

### Что означает copy-on-write?

Image layers доступны только для чтения.

При изменении файла контейнер записывает изменённую версию в свой верхний writable layer, не изменяя исходный image.
