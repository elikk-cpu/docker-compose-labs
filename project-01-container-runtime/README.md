# Проект 01. Контейнер как runtime-единица

**Сложность:** начальная  
**Оценка времени:** 3–5 часов

## Рабочая ситуация

Маркетинговой команде нужен временный preview-стенд со статическим сайтом.
Отдельный image пока собирать не требуется: контент и конфигурация передаются
с хоста.

## Цель

Поднять Nginx-контейнер, который:

- использует конкретный versioned tag;
- доступен с хоста на порту `8088`;
- доступен из другого контейнера по имени `landing`;
- получает сайт и конфиг через read-only bind mounts;
- находится в пользовательской bridge-сети;
- после удаления может быть воспроизведён одной документированной командой.

## Техническое задание

1. Скачай image, указанный в `.env.example`.
2. Создай пользовательскую bridge-сеть.
3. Запусти контейнер с понятным именем.
4. Подключи `site/` и `nginx/default.conf` в read-only режиме.
5. Правильно сопоставь host port и внутренний порт Nginx.
6. Проверь доступность с хоста.
7. Запусти отдельный диагностический curl-контейнер в той же сети и проверь DNS.
8. Сравни связность диагностического контейнера в bridge-сети и `none`.
9. Собери диагностику через `ps`, `logs`, `exec`, `inspect`, `history`, `stats`.
10. Воспроизведи:
    - мгновенное завершение контейнера из-за неверной команды;
    - неправильную публикацию порта.
11. Раздели два сценария:
    - `stop` + `start` — повторный запуск существующего контейнера;
    - `stop` + `rm` + новый `run` — пересоздание контейнера.
12. Покажи copy-on-write: создай временный файл во writable layer, удали
    контейнер и докажи, что файл исчез.

## Ограничения

- Нельзя использовать `latest`, `--privileged` или host network.
- Нельзя исправлять итоговое состояние ручным редактированием внутри контейнера.
- Нельзя публиковать дополнительные порты.
- Не записывай финальную команду только в history shell — добавь её в `RESULT.md`.

## Ты создаёшь

- `RESULT.md`;
- воспроизводимую команду запуска;
- краткое описание двух неисправностей;
- релевантные доказательства из `acceptance.md`.

## После выполнения мы научились

Работать с жизненным циклом контейнера, port publishing, read-only bind mounts,
Docker DNS, пользовательскими сетями и базовой runtime-диагностикой.

## Где пригодится

Preview-стенды, тестовые серверы, первичная диагностика недоступного контейнера
и подготовка к написанию Dockerfile.
